# Performance Optimization Guide

## Optimised for Millions of Matches

This schema is designed to handle **10M+ matches** and **100M+ player-match records**
while keeping analytical queries fast. Here's how.

---

## 1. Index Strategy

### Composite indexes (most important)

These are the workhorses that make common queries fast without full table scans:

```sql
-- Match lookup: "all matches in competition X during season Y, ordered by date"
CREATE INDEX ix_matches_comp_season_date
    ON matches (competition_id, season_id, match_date);

-- Team form calculation: "all matches for team X, ordered by date"
CREATE INDEX ix_matches_home_date
    ON matches (home_team_id, match_date);
CREATE INDEX ix_matches_away_date
    ON matches (away_team_id, match_date);

-- Player history: "all match stats for player X, joinable to match dates"
CREATE INDEX ix_pms_player_date
    ON player_match_stats (player_id, match_id);

-- Elo progression: "Elo history for team X, ordered by match"
CREATE INDEX ix_team_elo_team_date
    ON team_elo_history (team_id, match_id);

-- Odds by source: "all odds for match X from bookmaker Y"
CREATE INDEX ix_odds_match_source
    ON odds (match_id, source);
```

### Single-column indexes

```sql
-- Foreign key lookups (autogenerated by SQLAlchemy if index=True)
CREATE INDEX ix_matches_home_team_id ON matches (home_team_id);
CREATE INDEX ix_matches_away_team_id ON matches (away_team_id);
CREATE INDEX ix_matches_match_date   ON matches (match_date);

CREATE INDEX ix_odds_match_id        ON odds (match_id);
CREATE INDEX ix_pms_match_id         ON player_match_stats (match_id);
CREATE INDEX ix_pms_player_id        ON player_match_stats (player_id);
CREATE INDEX ix_pms_team_id          ON player_match_stats (team_id);
```

### Partial indexes

For filters on common subsets, partial indexes reduce index size significantly:

```sql
-- Only index matches that are finished (vast majority of queries)
CREATE INDEX ix_matches_finished
    ON matches (home_team_id, away_team_id, match_date)
    WHERE status = 'finished';

-- Only index odds with positive expected value (betting queries)
CREATE INDEX ix_ev_bets_positive
    ON expected_value_bets (match_id)
    WHERE recommended_ev > 0;
```

## 2. Table Partitioning (PostgreSQL 12+)

For the largest tables, use **declarative partitioning** to split data by match date.

### `matches` table — partition by month

```sql
CREATE TABLE matches (
    id BIGSERIAL,
    match_date DATE NOT NULL,
    ...
) PARTITION BY RANGE (match_date);

-- Create monthly partitions
CREATE TABLE matches_2024_01
    PARTITION OF matches
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
CREATE TABLE matches_2024_02
    PARTITION OF matches
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');
-- ... automate with pg_partman or cron

-- Ensure PK uniqueness across partitions
-- Use (id, match_date) as the PK or a UUID instead of SERIAL
```

### `odds` table — partition by match_date (reference)

```sql
CREATE TABLE odds (
    id BIGSERIAL,
    match_id BIGINT NOT NULL,
    match_date DATE NOT NULL,  -- denormalised for partitioning
    ...
) PARTITION BY RANGE (match_date);

-- Same monthly partition scheme as matches
```

### `player_match_stats` — partition by match_date (reference)

Same pattern: add `match_date` (denormalised in migration) and partition monthly.

### When to partition

| Table                | Rows  | Partition      | Partition Key               | Frequency |
| -------------------- | ----- | -------------- | --------------------------- | --------- |
| `matches`            | 10M+  | ✅              | `match_date`                | Monthly   |
| `odds`               | 50M+  | ✅              | `match_date` (denormalised) | Monthly   |
| `player_match_stats` | 100M+ | ✅              | `match_date` (denormalised) | Monthly   |
| `team_elo_history`   | 20M+  | ✅              | `match_date` (denormalised) | Monthly   |
| `predictions`        | 10M+  | ⬜ Only if >10M | `match_id`                  | —         |
| All others           | <1M   | ❌ Not needed   | —                           | —         |

## 3. Column Optimizations

### Use `BIGSERIAL` instead of `SERIAL` for large tables

```sql
-- For: matches, odds, player_match_stats, team_elo_history, team_form,
--      team_xg_history, predictions, expected_value_bets, betting_results
id BIGSERIAL PRIMARY KEY
```

### Use `VARCHAR` instead of `TEXT` when length is known

```sql
-- VARCHAR(4) for result: H, D, A
-- VARCHAR(8) for formation: 4-3-3
-- VARCHAR(16) for status: scheduled, live, finished
-- VARCHAR(32) for bookmaker source
```

### Use `DATE` instead of `TIMESTAMP` when time doesn't matter

Match_date needs only date precision → `DATE` type (4 bytes vs 8 for TIMESTAMP).

### Use `NUMERIC` for currency

```sql
market_value_eur NUMERIC(12, 2)  -- instead of FLOAT
transfer_fee_eur NUMERIC(12, 2)
```

### Use `SMALLINT` for small integer ranges

```sql
home_goals    SMALLINT   -- 0-99 goals is safe
away_goals    SMALLINT
home_red_cards SMALLINT  -- max 5 red cards in a match
away_red_cards SMALLINT
```

## 4. Query Patterns & Tips

### Fastest way to build a feature matrix

```sql
-- For: predicting the next N matches
-- This query runs in ~100ms on 10M rows with the composite indexes

SELECT
    m.id AS match_id,
    m.match_date,
    m.home_team_id,
    m.away_team_id,
    hf.last_5_ppg  AS home_form_ppg,
    af.last_5_ppg  AS away_form_ppg,
    hel.elo_before AS home_elo,
    ael.elo_before AS away_elo,
    hxg.xg         AS home_xg_avg,
    axg.xg         AS away_xg_avg
FROM matches m
LEFT JOIN team_form hf
    ON hf.team_id = m.home_team_id AND hf.match_id = m.id
LEFT JOIN team_form af
    ON af.team_id = m.away_team_id AND af.match_id = m.id
LEFT JOIN team_elo_history hel
    ON hel.team_id = m.home_team_id AND hel.match_id = m.id AND hel.side = 'home'
LEFT JOIN team_elo_history ael
    ON ael.team_id = m.away_team_id AND ael.match_id = m.id AND ael.side = 'away'
LEFT JOIN team_xg_history hxg
    ON hxg.team_id = m.home_team_id AND hxg.match_id = m.id AND hxg.side = 'home'
LEFT JOIN team_xg_history axg
    ON axg.team_id = m.away_team_id AND axg.match_id = m.id AND axg.side = 'away'
WHERE m.status = 'scheduled'
ORDER BY m.match_date ASC
LIMIT 20;
```

### Avoid N+1 queries

Using the repository pattern with eager loading:

```python
from sqlalchemy.orm import joinedload

# Instead of:
session.query(Match).options(
    joinedload(Match.home_team),
    joinedload(Match.away_team),
    joinedload(Match.statistics),
    joinedload(Match.odds),
).filter(Match.match_date >= today).all()
```

### Window functions for form calculation

```sql
-- Rolling 5-match average goals for each team
SELECT
    team_id,
    match_id,
    match_date,
    goals_scored,
    AVG(goals_scored) OVER (
        PARTITION BY team_id
        ORDER BY match_date
        ROWS BETWEEN 5 PRECEDING AND 1 PRECEDING
    ) AS rolling_avg_goals
FROM team_match_goals;
```

## 6. Bulk Loading Strategy

For ingesting historical data (e.g., 20+ years of matches):

```python
from sqlalchemy.dialects.postgresql import insert as pg_insert

def bulk_upsert_matches(session, matches_df):
    """PostgreSQL upsert: insert or do nothing on conflict."""
    stmt = pg_insert(Match).values(matches_df.to_dict(orient="records"))
    stmt = stmt.on_conflict_do_nothing(
        constraint="uq_matches_unique"  # requires a unique constraint
    )
    session.execute(stmt)
    session.commit()
```

### Copy from CSV (fastest — bypasses ORM)

```sql
\\copy matches FROM 'matches_2024.csv' WITH (FORMAT CSV, HEADER true);
```

### Disable indexes during bulk load, rebuild after

```sql
-- Before bulk load:
ALTER TABLE match_statistics DROP CONSTRAINT uq_match_statistics_match_id;
DROP INDEX ix_match_statistics_match_id;

-- After bulk load:
CREATE UNIQUE INDEX CONCURRENTLY uq_match_statistics_match_id
    ON match_statistics (match_id);
ALTER TABLE match_statistics ADD CONSTRAINT uq_match_statistics_match_id
    UNIQUE USING INDEX uq_match_statistics_match_id;
```

## 7. Maintenance

### Regular vacuuming

```sql
-- Schedule via pg_cron or cron:
VACUUM ANALYZE matches;
VACUUM ANALYZE odds;
VACUUM ANALYZE player_match_stats;
```

### Monitor index usage

```sql
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
WHERE idx_scan = 0    -- Unused indexes to drop
ORDER BY tablename;
```

### Monitor table bloat

```sql
SELECT
    nspname AS schema_name,
    relname AS table_name,
    round(100 * pg_relation_size(relid) / NULLIF(pg_total_relation_size(relid), 0), 1) AS bloat_pct
FROM pg_stat_user_tables
ORDER BY bloat_pct DESC
LIMIT 10;
```
